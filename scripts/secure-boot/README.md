# 🔐 SecureBoot Scripts

Complete guide for SecureBoot key management, enrollment, and enablement operations.

## 📚 Overview

This directory contains scripts for managing SecureBoot on **physical host machines**. For QEMU/VM operations, see `scripts/qemu/`.

### Key Distinction: Host vs QEMU

- **Host Operations** (this directory): Work on your physical machine's UEFI firmware
- **QEMU Operations** (`scripts/qemu/`): Work within QEMU virtual machines for testing

## 🔑 Key Generation

### `generate-sb-keys.sh`
Generate complete SecureBoot key hierarchy (PK, KEK, db).

```bash
# Via pf task
./pf.py secure-keygen

# Or directly
./scripts/secure-boot/generate-sb-keys.sh
```

**Creates**:
- `keys/PK.key`, `PK.crt`, `PK.cer` - Platform Key (root of trust)
- `keys/KEK.key`, `KEK.crt`, `KEK.cer` - Key Exchange Key
- `keys/db.key`, `db.crt`, `db.cer` - Signature Database

**Output**: Keys in `keys/` directory (RSA-4096, 10 years validity)

### `create-auth-files.sh`
Create authenticated enrollment files (ESL and AUTH formats).

```bash
# Via pf task
./pf.py secure-make-auth

# Or directly
./scripts/secure-boot/create-auth-files.sh
```

**Requires**: Keys generated by `generate-sb-keys.sh`

### `der-extract.sh`
Extract DER/PKCS#12 certificate bundles into separate PEM files.

```bash
# Via pf task
PATH=/path/to/bundle.der OUT_DIR=out/keys NAME=MyKey ./pf.py secure-der-extract

# Or directly
./scripts/secure-boot/der-extract.sh /path/to/bundle.der out/keys MyKey
```

## 📥 Key Enrollment

### `enroll-secureboot.sh`
Enroll custom SecureBoot keys into OVMF (QEMU firmware).

```bash

# Or directly
./scripts/secure-boot/enroll-secureboot.sh
```

**Use case**: Automated key enrollment for QEMU testing

### `enroll-secureboot-nosudo.sh`
Alternative enrollment method without sudo requirements.


## 🔍 Status and Verification

### `check-secureboot-status.sh`
Check SecureBoot status and system capability.

```bash
# Via pf task (recommended)
./pf.py secureboot-check

# Or directly
./scripts/secure-boot/check-secureboot-status.sh
```

**Checks**:
- UEFI system compatibility
- Current SecureBoot state (enabled/disabled/unsupported)
- Setup Mode status
- Kernel configuration for firmware access
- Kexec availability
- Available enablement methods

**Output**: Detailed status report with recommendations

## ⚙️ SecureBoot Enablement (Advanced)

### `enable-secureboot-kexec.sh`
**[HOST ONLY]** Enable SecureBoot on physical machine via double kexec method.

```bash
# Via pf task (recommended)
sudo ./pf.py secureboot-enable-host-kexec

# Or directly
sudo ./scripts/secure-boot/enable-secureboot-kexec.sh
```

**⚠️ IMPORTANT**: This is a **framework implementation** demonstrating the double kexec workflow. Actual SecureBoot enablement requires hardware-specific code or BIOS setup.

**What it does**:
1. Checks prerequisites (kexec, alternate kernels)
2. Prepares alternate kernel with relaxed security
3. Framework for switching kernels without reboot
4. Provides structure for hardware-specific enablement

**Recommended Approach**: Use BIOS/UEFI setup to enable SecureBoot (safest and most reliable).

## 🎯 Common Workflows

### Workflow 1: Generate SecureBoot Keys for Bootable Media

```bash
# 1. Generate keys
./pf.py secure-keygen

# 2. Create authentication files
./pf.py secure-make-auth

# 3. Create bootable media
ISO_PATH=/path/to/ubuntu.iso ./pf.py secureboot-create
```

### Workflow 2: Check SecureBoot Status

```bash
# Quick status check
./pf.py secureboot-check

# View enrollment state
mokutil --sb-state
efibootmgr -v
```

### Workflow 3: Custom Key Generation for Signing

```bash
# Generate custom keys
./scripts/secure-boot/sb_keys_custom.sh

# Sign bootloader
sbsign --key keys/db.key --cert keys/db.crt --output signed.efi original.efi
```

## 🔧 Centralized Key Management

### `keys-centralize.sh`
Move keys from legacy locations to centralized `out/keys/` structure.

```bash
# Via pf task
./pf.py secure-keys-centralize

# With cleanup (backup + remove legacy locations)
./pf.py secure-keys-prune
```

**Organizes**:
- `out/keys/PK/` - Platform Keys
- `out/keys/KEK/` - Key Exchange Keys
- `out/keys/db/` - Signature Database keys
- `out/keys/mok/` - Machine Owner Keys

## 🚫 Common Pitfalls

### 1. Confusing Host vs QEMU Operations

**Problem**: Running QEMU scripts on host or vice versa

**Solution**: 
- Scripts in `scripts/secure-boot/` are for **physical machines**
- Scripts in `scripts/qemu/` are for **virtual machines**
- Task names now include `[HOST]` or `[QEMU]` prefixes

### 2. Enabling SecureBoot Without Keys

**Problem**: Trying to enable SecureBoot before generating keys

**Solution**: Always generate and enroll keys first:
```bash
./pf.py secure-keygen
./pf.py secure-make-auth
# Then enable SecureBoot in BIOS setup
```

### 3. Using enable-secureboot-kexec.sh Expecting Full Automation

**Problem**: Script doesn't actually enable SecureBoot

**Solution**: This is a framework. Use BIOS/UEFI setup for reliable enablement. The script demonstrates the workflow for advanced users who want to implement hardware-specific enablement.

## 🔒 Security Notes

1. **Private Key Protection**
   - All `.key` files should have 600 permissions
   - Never commit private keys to version control
   - Store backups securely (encrypted)

2. **Key Hierarchy**
   - PK is the root of trust - protect it most carefully
   - KEK and db can be rotated if needed
   - Loss of PK means loss of SecureBoot control

3. **Signing Best Practices**
   - Always verify signatures after signing
   - Use consistent keys across your infrastructure
   - Document which keys are used where
